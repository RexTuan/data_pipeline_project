# **現代數據工程實戰：基於雲端與容器化技術的30日全端管線建置指南**

## **系統規劃藍圖**

在數據驅動的現代企業環境中，資料基礎架構的穩定性與自動化能力是決定分析洞察品質的核心基石。對於具備豐富業務邏輯與分析敏銳度的專業人員而言，跨足系統底層的建置與維護，不僅是技術棧的擴張，更是思維模式的根本轉變。傳統的分析思維專注於「如何從現有數據中提取價值與模型準確度」，而工程思維則必須嚴謹地思考「如何打造具備容錯性、可重現性且自動化的系統，確保數據在不同節點間穩定流動」1。

本指南旨在提供一份詳盡、具備實戰深度的專案規劃與 30 日上手教學手冊。系統架構完全依循現代資料堆疊（Modern Data Stack）的最佳實踐，將資料的生命週期拆解為獨立運作且低耦合的模組。專案情境設定為自動化金融數據管線，整體流程包含：透過外部 API 擷取每日市場交易資料，載入本地端關聯式資料庫進行過渡儲存，經由自動化工作流排程引擎轉換並上傳至雲端資料倉儲，最終結合最新的生成式人工智慧模型進行自動化摘要分析，並透過即時通訊軟體發送系統日誌與營運報告。

### **核心技術與產品名詞定義**

為確保系統規劃的清晰度，本報告在探討各模組時，將針對關鍵技術名詞與產品進行嚴謹的定義與應用舉例：

* **(名詞解釋：Data Analyst / 數據分析師)**：專注於解釋數據、建立儀表板與機器學習模型，以解決商業問題並提供決策支援的專業人員。應用舉例：利用 SQL 查詢資料庫，並將結果匯出至 Tableau 製作營收趨勢報表2。  
* **(名詞解釋：Data Engineering / 數據工程)**：建立、維護與最佳化資料流動的基礎架構，確保龐大資料集能可靠、安全且高效地被處理的工程學科。應用舉例：構建自動化 ETL 管線，為下游的分析師提供乾淨、去重且即時的數據來源4。  
* **(名詞解釋：FinMind API)**：提供豐富台灣金融與總體經濟開源數據的應用程式介面，涵蓋股價、財報及三大法人等超過 50 種數據。應用舉例：開發 Python 爬蟲腳本呼叫該 API，自動獲取台積電每日的開盤、收盤價及交易量6。  
* **(名詞解釋：Docker)**：一種輕量級的容器化技術，將應用程式與其執行所需的作業系統依賴庫打包成標準化單元，確保跨平台執行的一致性。應用舉例：開發者可透過一份設定檔，在任何作業系統上一鍵同時啟動資料庫與排程器，免除繁瑣的環境依賴配置8。  
* **(名詞解釋：Docker Compose)**：用於定義和執行多容器 Docker 應用程式的工具，透過 YAML 檔案配置應用程式的各項服務。應用舉例：在單一 docker-compose.yaml 中同時宣告 PostgreSQL 與 Airflow 服務，並配置兩者間的內部網路通訊10。  
* **(名詞解釋：PostgreSQL)**：一款功能強大、高度開源的物件關聯式資料庫管理系統（ORDBMS），以其高可靠性、資料完整性及支援複雜查詢著稱。應用舉例：在資料管線中作為本地端的中介儲存層（Staging Area），利用其進階語法快速處理股票資料的去重12。  
* **(名詞解釋：Apache Airflow)**：由開源社群維護的工作流編排平台，透過 Python 程式碼定義、排程及監控資料管線。應用舉例：設定每週一至週五下午自動觸發抓取當日收盤價的任務，若失敗則自動重試並發送警報9。  
* **(名詞解釋：DAG / 有向無環圖)**：在 Airflow 中代表資料管線的結構，由多個任務（節點）與執行相依性（有向邊）組成，且不允許產生無限迴圈。應用舉例：嚴格定義任務順序，例如「必須先成功完成資料庫寫入任務，才能啟動雲端上傳任務」9。  
* **(名詞解釋：Google BigQuery Sandbox)**：Google Cloud 提供的全代管企業級資料倉儲，其沙盒模式允許使用者在免綁定信用卡的情況下進行數據分析。應用舉例：將歷年海量的股票交易明細匯入其中，使用標準 SQL 在數秒內完成跨年度的移動平均線計算16。  
* **(名詞解釋：Gemini API)**：Google 推出的多模態大型語言模型應用程式介面，具備強大的文字生成、理解與邏輯推理能力。應用舉例：將資料倉儲算出的財務數字轉換為文字提示，交由模型自動生成如「本季營收成長顯著」的商業摘要報告18。  
* **(名詞解釋：Discord Webhook)**：一種允許外部系統透過 HTTP POST 請求，將自動化訊息與格式化內容直接推送到指定通訊頻道的機制。應用舉例：當系統捕捉到資料載入失敗的錯誤時，立即將錯誤代碼與時間戳記發送至工程團隊的除錯頻道20。  
* **(名詞解釋：Idempotency / 冪等性)**：無論一項操作被執行一次還是多次，系統的最終狀態都保持一致，不會產生重複或錯誤的副作用。應用舉例：若排程因網路中斷而重試執行同一天的資料寫入任務，資料庫內絕對不會出現兩筆相同的股價紀錄1。  
* **(名詞解釋：Upsert)**：資料庫操作中的一種混合指令（Update \+ Insert），當記錄不存在時插入新資料，若主鍵衝突則更新現有資料。應用舉例：使用 PostgreSQL 的 ON CONFLICT DO UPDATE 語法，確保每日股票資料的唯一性與正確性12。

### **架構與資料流向設計**

本專案採用的架構將最大化使用免費與開源軟體。資料流動的生命週期如下：

| 階段 | 負責組件 | 執行動作與技術細節 |
| :---- | :---- | :---- |
| **資料提取 (Extract)** | Python 腳本 & FinMind API | 定期向 API 請求特定股票的日K線與財務數據。 |
| **本地中介 (Stage)** | Docker 內的 PostgreSQL | 作為 ODS（Operational Data Store），進行初步的資料清洗與 Upsert 操作。 |
| **轉換與載入 (Load)** | Python 腳本 (google-cloud-bigquery) | 將關聯式資料庫中的資料批次匯出，並載入至 Google BigQuery。 |
| **管線編排 (Orchestrate)** | Docker 內的 Apache Airflow | 管理上述所有任務的相依性，並設定基於台灣時區的 Cron 排程。 |
| **智能分析 (Analyze)** | Python 腳本 & Gemini API | 自 BigQuery 讀取聚合後的數據，呼叫 Gemini 2.5 Flash 生成文字洞察。 |
| **監控警報 (Notify)** | Discord Webhook | 將執行結果、AI 報告或中斷錯誤封裝為 JSON 結構推播至頻道。 |

此架構展現了現代數據工程的幾個核心理念：基礎架構即程式碼（透過 Docker）、模組化（每個處理步驟獨立為一個 Airflow Task）、以及雲端原生擴展性（資料最終落地於 BigQuery）。

3. **建立專案結構**：建立名為 data\_pipeline\_project 的資料夾。為了讓 Airflow 容器能與本地主機共享資料而不發生權限衝突，必須建立特定的目錄並設定環境變數：

#### **Day 3 \- Day 4: PostgreSQL 部署與資料建模 (Data Modeling)**

資料庫的綱要（Schema）設計決定了後續查詢的效率。

1. **撰寫 docker-compose 檔案**：在專案根目錄建立 docker-compose.yaml，初步定義 PostgreSQL 服務。這將作為 Airflow 的後台元資料庫（Metadata Database），同時充當本專案的中介儲存層23。  
   YAML  
   version: '3.8'  
   services:  
     postgres:  
       image: postgres:16  
       environment:  
         POSTGRES\_USER: airflow  
         POSTGRES\_PASSWORD: airflow  
         POSTGRES\_DB: airflow  
       volumes:  
         \- postgres-db-volume:/var/lib/postgresql/data  
       ports:  
         \- "5432:5432"  
   volumes:  
     postgres-db-volume:

   執行 docker compose up \-d postgres，即可在背景啟動一個原生的 PostgreSQL 16 實例。  
2. **設計資料表與索引**： (名詞解釋：B-Tree Index / B樹索引 \- 一種資料結構，能讓資料庫以對數時間複雜度快速搜尋、排序與檢索特定範圍的值。應用舉例：在依日期查詢特定股票時，避免資料庫進行緩慢的全表掃描24。) 使用 DBeaver 等客戶端工具連接至 localhost:5432，建立股票資料表。考量到資料的唯一性與查詢模式，我們將 stock\_id 與 trade\_date 設為複合主鍵（Composite Primary Key）：  
   SQL  
   CREATE TABLE IF NOT EXISTS tw\_stock\_daily (  
       stock\_id VARCHAR(10) NOT NULL,  
       trade\_date DATE NOT NULL,  
       open\_price NUMERIC,  
       high\_price NUMERIC,  
       low\_price NUMERIC,  
       close\_price NUMERIC,  
       trade\_volume BIGINT,  
       PRIMARY KEY (stock\_id, trade\_date)  
   );

   建立主鍵的同時，PostgreSQL 會自動在底層建立 B-Tree 索引，這對於未來 Airflow 依據時間區間（如 WHERE trade\_date \= '2026-03-01'）進行資料提取時，具有決定性的效能提升24。

#### **Day 5 \- Day 7: API 串接與防呆寫入 (Idempotent Upsert)**

此環節將實作資料提取腳本。FinMind 提供了穩定的台灣金融數據，免費用戶的 API 請求上限為每小時 300 次，若至官網註冊並取得 Token，可將上限提升至 600 次/小時6。

1. **開發資料擷取模組**：在 scripts/ 目錄下建立 fetch\_stock.py。  
2. **實踐冪等性寫入**：在傳統分析腳本中，若資料寫入失敗，分析師可能會手動清空資料表後重跑。但在自動化工程中，必須採用 ON CONFLICT DO UPDATE（Upsert）語法。這保證了當 Airflow 觸發重試機制時，同一天的資料不會被重複插入（Duplicate Records），而是直接更新現有數值22。  
   Python  
   import psycopg2  
   import requests

   def fetch\_and\_upsert\_stock(stock\_id, target\_date):  
       \# 1\. 呼叫 FinMind API  
       url \= f"https://api.finmindtrade.com/api/v4/data?dataset=TaiwanStockPrice\&data\_id={stock\_id}\&start\_date={target\_date}\&end\_date={target\_date}"  
       response \= requests.get(url).json()  
       data\_list \= response.get('data',)

       if not data\_list:  
           print(f"No data found for {stock\_id} on {target\_date}")  
           return

       \# 2\. 建立資料庫連線  
       conn \= psycopg2.connect(host="postgres", database="airflow", user="airflow", password="airflow")  
       cur \= conn.cursor()

       \# 3\. 定義 Upsert SQL  
       upsert\_sql \= """  
           INSERT INTO tw\_stock\_daily (stock\_id, trade\_date, open\_price, high\_price, low\_price, close\_price, trade\_volume)  
           VALUES (%s, %s, %s, %s, %s, %s, %s)  
           ON CONFLICT (stock\_id, trade\_date)  
           DO UPDATE SET  
               open\_price \= EXCLUDED.open\_price,  
               high\_price \= EXCLUDED.high\_price,  
               low\_price \= EXCLUDED.low\_price,  
               close\_price \= EXCLUDED.close\_price,  
               trade\_volume \= EXCLUDED.trade\_volume;  
       """  
       \# 4\. 寫入資料庫  
       for row in data\_list:  
           cur.execute(upsert\_sql, (  
               row\['stock\_id'\], row\['date'\], row\['open'\],  
               row\['max'\], row\['min'\], row\['close'\], row  
           ))

       conn.commit()  
       cur.close()  
       conn.close()

   此處的 %s 參數化查詢能有效防止 SQL 注入攻擊，同時 EXCLUDED 關鍵字能直接引用衝突發生時原本企圖插入的新值，為業界處理高頻更新資料的標準實踐12。

### **第二階段：雲端資料倉儲與轉載架構（Day 8 \- Day 15）**

關聯式資料庫適合處理逐筆交易（OLTP），但當涉及跨年度的大規模數據分析時，必須依賴雲端資料倉儲。此階段將引導您配置 Google BigQuery，並建立自動化資料載入機制。

#### **Day 8 \- Day 10: Google BigQuery Sandbox 啟用與權限配置**

1. **開通沙盒模式**：使用 Google 帳號登入 Google Cloud Console。BigQuery Sandbox 允許使用者在不提供信用卡資訊、不建立計費帳戶的狀態下探索強大的運算能力17。  
2. **沙盒資源限制認知**：

#### **Day 11 \- Day 15: Python 批次載入腳本開發**

在將資料從 PostgreSQL 遷移至 BigQuery 時，Python 生態系提供了兩個主要的開源套件：pandas-gbq 與 google-cloud-bigquery。

| 特性比較 | pandas-gbq | google-cloud-bigquery |
| :---- | :---- | :---- |
| **底層維護** | PyData 與開源社群志工34 | Google 官方維護的 Python 客戶端庫34 |
| **使用場景** | 輕量級的資料框讀寫，適合分析師快速驗證34 | 生產環境的資料工程，支援管理資料表、上傳 JSON/Parquet 格式檔案34 |
| **效能與控制** | 預設以 CSV 序列化寫入，對時間格式支援佳，但缺乏精細的作業配置35 | 支援設定 QueryJobConfig，並可透過 Parquet 格式大幅加速百萬級資料上傳34 |

為了符合生產環境標準，本專案採用 google-cloud-bigquery 客戶端庫搭配 load\_table\_from\_dataframe 方法進行資料同步36。 建立 scripts/pg\_to\_bq.py 腳本：

Python

import os  
import pandas as pd  
from sqlalchemy import create\_engine  
from google.cloud import bigquery

\# 設定 GCP 認證環境變數  
os.environ \= "/path/to/your-service-account.json"

def sync\_postgres\_to\_bq():  
    \# 1\. 從 PostgreSQL 讀取最新資料  
    db\_uri \= "postgresql+psycopg2://airflow:airflow@postgres:5432/airflow"  
    engine \= create\_engine(db\_uri)  
    \# 僅提取近期資料以節省運算成本  
    query \= "SELECT \* FROM tw\_stock\_daily WHERE trade\_date \>= CURRENT\_DATE \- INTERVAL '7 days'"  
    df \= pd.read\_sql(query, engine)

    if df.empty:  
        print("No new data to sync.")  
        return

    \# 2\. 初始化 BigQuery 客戶端  
    client \= bigquery.Client()  
    table\_id \= f"{client.project}.stock\_analytics.tw\_stock\_daily"

    \# 3\. 設定載入作業組態  
    job\_config \= bigquery.LoadJobConfig(  
        write\_disposition="WRITE\_TRUNCATE", \# 覆寫目標表，解決沙盒不支援 DML 語句的限制  
        autodetect=True \# 自動推斷資料型態  
    )

    \# 4\. 執行載入作業  
    job \= client.load\_table\_from\_dataframe(df, table\_id, job\_config=job\_config)  
    job.result()  \# 阻塞等待作業完成  
    print(f"✅ 成功同步 {job.output\_rows} 筆資料至 BigQuery.")

由於 BigQuery 沙盒模式不支援 UPDATE 與 DELETE 等 DML 操作17，傳統的 Merge/Upsert 無法在沙盒中執行。因此，我們在 LoadJobConfig 中使用 WRITE\_TRUNCATE（完全覆寫）策略，確保每次載入的資料狀態皆與來源端保持一致，這是在資源受限環境下達成系統冪等性的常見工程折衷方案22。

### **第三階段：Apache Airflow 排程與自動化編排（Day 16 \- Day 22）**

此階段是整個專案的神經中樞。我們將部署 Apache Airflow，將前述的獨立腳本串聯為具備依賴關係與重試機制的資料管線（Pipeline）。

#### **Day 16 \- Day 18: Airflow 環境客製化與時區設定**

1. **擴展 Docker Compose 檔案**：下載官方的 docker-compose.yaml 後，必須進行客製化以符合本專案需求。 (名詞解釋：Dockerfile \- 一個文字文件，包含了一系列讓 Docker 引擎自動構建映像檔的指令。應用舉例：在官方 Airflow 映像檔的基礎上，透過 Dockerfile 追加安裝 pandas 與 google-cloud-bigquery 函式庫8。) 建立 Dockerfile：  
   Dockerfile  
   FROM apache/airflow:3.1.7  
   COPY requirements.txt /  
   RUN pip install \--no-cache-dir \-r /requirements.txt

   在 docker-compose.yaml 中，將 image: apache/airflow:3.1.7 註解掉，改為 build:.，指示 Docker 讀取本地的 Dockerfile 進行構建8。  
2. **時區配置（Time Zone Configuration）**： 台灣股票市場具有嚴格的交易時間（09:00 \- 13:30），若排程器依賴預設的 UTC 時間，將極易造成維運上的混亂。Airflow 支援將系統與 UI 切換為本地時區40。 在 .env 檔案中加入：  
   程式碼片段  
   AIRFLOW\_\_CORE\_\_DEFAULT\_TIMEZONE=Asia/Taipei  
   AIRFLOW\_\_WEBSERVER\_\_DEFAULT\_UI\_TIMEZONE=Asia/Taipei

   此外，在自訂的 Dockerfile 中，也可寫入環境變數以確保底層 OS 時區一致41：  
   Dockerfile  
   ENV TZ=Asia/Taipei  
   RUN ln \-snf /usr/share/zoneinfo/$TZ /etc/localtime && echo $TZ \> /etc/timezone

3. **系統初始化**：執行 docker compose build 構建映像檔，隨後執行 docker compose up airflow-init 進行後台資料庫遷移。完成後以 docker compose up \-d 啟動所有服務（Webserver 預設運行於 port 8080）8。

#### **Day 19 \- Day 22: 撰寫與部署 DAG (Directed Acyclic Graph)**

開發 Airflow DAG 時，必須定義任務（Tasks）以及其執行拓撲（Topology）。我們將採用 Pendulum 函式庫來精準控制 DAG 的排程時間43。

在 dags/ 資料夾下建立 tw\_stock\_pipeline.py：

Python

import pendulum  
from datetime import timedelta  
from airflow import DAG  
from airflow.operators.python import PythonOperator

\# 匯入自定義模組  
from scripts.fetch\_stock import fetch\_and\_upsert\_stock  
from scripts.pg\_to\_bq import sync\_postgres\_to\_bq

\# 定義本地時區  
local\_tz \= pendulum.timezone("Asia/Taipei")

\# 定義 DAG 的預設參數  
default\_args \= {  
    'owner': 'data\_engineer',  
    'depends\_on\_past': False,  
    'retries': 3, \# 失敗時最多重試 3 次  
    'retry\_delay': timedelta(minutes=5), \# 每次重試間隔 5 分鐘  
}

with DAG(  
    dag\_id='daily\_tw\_stock\_pipeline',  
    default\_args=default\_args,  
    description='Fetch stock data, store in Postgres, and sync to BigQuery',  
    start\_date=pendulum.datetime(2026, 3, 1, tz=local\_tz),  
    schedule\_interval='30 15 \* \* 1-5', \# 台灣時間每週一至週五下午 15:30 執行  
    catchup=False, \# 關閉回溯執行，避免啟動時瞬間產生大量請求  
    max\_active\_runs=1  
) as dag:

    \# 任務一：擷取並寫入本地資料庫  
    extract\_task \= PythonOperator(  
        task\_id='extract\_and\_upsert\_to\_pg',  
        python\_callable=fetch\_and\_upsert\_stock,  
        op\_kwargs={  
            'stock\_id': '2330',   
            'target\_date': '{{ ds }}' \# Jinja Template，動態取得執行的邏輯日期 (YYYY-MM-DD)  
        }  
    )

    \# 任務二：同步至雲端資料倉儲  
    load\_bq\_task \= PythonOperator(  
        task\_id='sync\_pg\_to\_bq',  
        python\_callable=sync\_postgres\_to\_bq  
    )

    \# 定義依賴關係：必須擷取成功才能進行同步  
    extract\_task \>\> load\_bq\_task

(名詞解釋：Jinja Template \- 一種模板引擎，在 Airflow 中用來在運行時動態注入變數。應用舉例：使用 {{ ds }} 自動將當天的日期字串傳遞給 Python 函數，免去硬編碼日期的麻煩45。)

完成後，進入 Airflow Web UI，將該 DAG 的開關打開（Unpause）。您可以觀察任務節點如何依序變為綠色（Success），並學習點擊節點檢視 Logs 進行問題排除（Troubleshooting）46。

#### **Day 23 \- Day 26: 串接 Gemini 2.5 Flash 進行智能總結**

Google 於 2026 年初大幅調整了 Gemini API 的限制，為適應高吞吐量應用，推出了 Gemini 2.5 Flash 系列模型。在免費用戶（Free Tier）層級下，該模型提供最高 15 RPM（每分鐘請求數）及 1,000 RPD（每日請求數）的配額，並具備 100 萬 Token 的超大上下文視窗，非常適合用來解析數值型報表48。

1. **安裝官方 SDK**：在 requirements.txt 內加入 google-genai 並重新構建 Docker 映像檔19。  
2. **設計 Prompt 與分析腳本 (scripts/ai\_analysis.py)**：  
   資料工程師需透過程式化方式組合數據並傳遞給模型：  
   Python  
   import os  
   import pandas as pd  
   from sqlalchemy import create\_engine  
   from google import genai

   def generate\_ai\_summary():  
       \# 1\. 取得近期分析數據  
       engine \= create\_engine("postgresql+psycopg2://airflow:airflow@postgres:5432/airflow")  
       df \= pd.read\_sql("SELECT trade\_date, close\_price, trade\_volume FROM tw\_stock\_daily ORDER BY trade\_date DESC LIMIT 5", engine)

       \# 將 DataFrame 轉為文字格式  
       data\_text \= df.to\_string(index=False)

       \# 2\. 呼叫 Gemini 2.5 Flash 模型  
       client \= genai.Client(api\_key=os.environ.get("GEMINI\_API\_KEY"))  
       prompt \= f"""  
       你是一位專業的金融量化分析師。請分析以下台積電(2330)近五日的股價與交易量數據。  
       請提供一份約 150 字的繁體中文分析報告，點出價格趨勢與量價關係的異常變化：

       {data\_text}  
       """

       response \= client.models.generate\_content(  
           model='gemini-2.5-flash',  
           contents=prompt,  
       )  
       return response.text

   此步驟利用了大型語言模型強大的上下文理解能力，將冰冷的 BigQuery 儲存結果轉換為對商業決策有價值的敘述性報告18。

#### **Day 27 \- Day 29: 整合 Discord Webhook 通知系統**

良好的管線需要具備可觀測性（Observability）。當任務成功並產出 AI 報告，或當管線發生例外崩潰時，系統應主動通知維運人員，而非被動等待分析師發現儀表板未更新22。

1. **建立 Webhook URL**：在 Discord 伺服器設定中新增一組 Webhook，並複製其 URL51。  
2. **開發通知發送模組 (scripts/discord\_notifier.py)**： Discord 的 API 嚴格要求接收特定格式的 JSON Payload（酬載）21。  
   Python  
   import requests  
   import json

   def send\_discord\_msg(content\_text):  
       webhook\_url \= "https://discord.com/api/webhooks/YOUR\_WEBHOOK\_ID/YOUR\_TOKEN"

       \# 建構符合 Discord 規範的 JSON 結構  
       payload \= {  
           "username": "Data Pipeline Bot",  
           "avatar\_url": "https://example.com/bot-icon.png",  
           "embeds": \[  
               {  
                   "title": "📊 每日資料管線執行報告",  
                   "description": content\_text,  
                   "color": 3447003 \# 藍色  
               }  
           \]  
       }

       headers \= {"Content-Type": "application/json"}  
       response \= requests.post(webhook\_url, data=json.dumps(payload), headers=headers)

       if response.status\_code not in (200, 204):  
           print(f"Webhook發送失敗，HTTP狀態碼: {response.status\_code}")

3. **整合至 Airflow DAG**： 將上述的 AI 分析與通知函數封裝為一個 PythonOperator，並將其設定為管線的最後一個節點（load\_bq\_task \>\> ai\_notify\_task）。此外，為了捕捉錯誤，可在 default\_args 中加入 on\_failure\_callback 參數，指定當任何任務拋出例外時，自動呼叫 Discord 發送「🔴 系統警報：任務執行失敗」的通知51。
